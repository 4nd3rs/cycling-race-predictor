# Send Notifications — Policy & Memory

## Identity
You send personalized race notifications to users via Telegram and WhatsApp.
Messages are generated by Gemini (flash-lite) using cycling-specific tone and context.

## Trigger Windows
- **Preview** (T-36h to T-60h): race coming up in ~2 days
- **Raceday** (T-0h to T-6h): race morning, short punchy message
- **Result** (T+0h to T+12h): race finished with results in DB

## Message Generation
- Uses Gemini 2.5 Flash Lite via API
- Prompt includes: race context, top 5 predictions, user's followed riders, discipline
- Tone: knowledgeable cycling analyst, first person, European, no hype
- Telegram gets HTML formatting (bold first line), WhatsApp gets plain text

## Channels
- **Telegram**: Bot API sendMessage with HTML parse_mode
- **WhatsApp**: Twilio API, plain text only
- No image attachments on Vercel (text only)

## Dedup & Frequency
- Check `notification_log` before sending (userId + raceId + messageType + channel)
- Log after successful send
- Frequency gates per message type:
  - preview: all, key-moments
  - raceday: all
  - result: all, key-moments, race-day-only

## Quiet Hours
- Skip entirely if UTC hour < 7 or > 22

## Concurrency
- Batch sends with Promise.allSettled, max 20 concurrent tasks per batch

## Schedule
- Runs every 2 hours: `0 */2 * * *`

## Failure Handling
- Individual send failures are non-fatal (logged, counted)
- Gemini API failures → skip that user (no retry)
- Route returns JSON summary with sent/skipped/dupes counts
